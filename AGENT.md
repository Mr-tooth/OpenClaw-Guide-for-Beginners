## 📋 项目概览
### 项目背景
OpenClaw是当前最热门的开源自托管AI Agent网关，支持10+消息平台对接、1500+技能扩展，拥有大量中文用户。但官方文档对新手不友好，中文优质教程稀缺，用户安装配置过程中踩坑率极高。

### 项目目的
打造一站式中文新手教程，覆盖从安装、配置、平台对接、进阶使用全流程，降低中文用户使用OpenClaw的门槛，成为中文用户首选的入门参考资料。

### 目标受众
- 纯新手用户：完全没接触过OpenClaw、甚至没接触过Linux/命令行的用户
- 技术爱好者：有一定基础，想快速部署OpenClaw的用户
- 开发者：想二次开发OpenClaw、定制技能的用户

### 当前状态
- 版本: v1.6.0
- 对齐OpenClaw版本: v2026.4.29（最新稳定版）
- 内容覆盖: 全平台安装教程、API配置、主流平台对接、进阶使用
- 已完成目录结构重构，所有链接验证有效，无死链

## 🌟 核心开发原则
### 1. 新手友好优先
- 所有步骤必须详细，禁止跳步，默认用户是纯新手
- 专业术语必须解释，比如提到「WSL」要说明是「Windows子系统Linux」
- 复杂操作要分步骤，每一步要有明确的预期输出说明
- 避免使用「简单」「容易」「只需」等容易给用户造成压力的词汇

### 2. 准确性第一
- 所有命令、配置、链接必须100%验证有效，禁止提供未测试的内容
- 涉及系统操作的命令必须加风险提示，比如`sudo`、`rm`操作
- 时效性内容必须标注有效期，比如「2026年5月有效」
- 所有版本信息必须对齐OpenClaw官方最新稳定版

### 3. 一致性原则
- 术语统一：统一使用「OpenClaw」（首字母大写），禁止写「openclaw」「Openclaw」
- 格式统一：所有文档格式、标题层级、代码块风格保持一致
- 链接统一：内部链接全部使用相对路径，禁止使用绝对路径
- 风格统一：保持客观、简洁、实用的写作风格，避免情绪化表述

### 4. 安全性原则
- 禁止提供可能损坏用户系统的命令，比如`rm -rf /`这类危险命令
- 所有涉及用户数据的操作必须加备份提示
- 涉及公网暴露的配置必须加安全提示，比如「请配置防火墙避免暴露端口」
- 禁止在文档中泄露用户敏感信息、密钥等内容

### 5. 时效性原则
- API平台优惠政策、版本变更信息要及时更新
- 过期内容要标注「已过时」，或者直接删除，避免误导用户
- 所有更新必须在CHANGELOG.md中记录更新时间和内容

## 📝 开发规范
### Markdown格式规范
```markdown
# 一级标题（仅页面顶部使用）
## 二级标题（主要章节）
### 三级标题（小节）
#### 四级标题（子项，尽量少用）

- 无序列表使用短横线
1. 有序列表使用数字加点

```bash
# 代码块必须标注语言（bash/powershell/json等）
echo "Hello World"
```

> 引用块用于提示、警告类内容

| 表格列1 | 表格列2 |
|---------|---------|
| 内容1   | 内容2   |
```

### 链接规范
- 内部链接使用相对路径：比如`[快速上手](docs/getting-started/quickstart.md)`
- 外部链接必须包含完整https协议：比如`[官网](https://openclaw.ai)`
- 所有链接必须验证有效，禁止出现死链
- 作者的邀请链接必须保持原样，禁止修改、替换

### 内容更新规范
1. 每次更新必须在CHANGELOG.md中记录：
   ```markdown
   ## [v1.6.1] - 2026-05-03
   - 修复Windows教程中链接错误
   - 更新火山引擎优惠政策
   ```
2. 时效性内容必须标注更新时间
3. 大版本更新（比如对齐OpenClaw新版本）必须在README中更新版本号
4. 涉及目录结构变更必须同步更新README中的目录说明

### PR提交规范
- 每次PR只能做一件事：比如「修复Windows链接错误」「新增QQ对接教程」
- PR标题必须清晰说明变更内容，禁止模糊描述
- 所有变更必须本地验证通过后才能提交PR
- 禁止在PR中包含无关变更

## 🎯 常见任务指引
### 任务1：修复bug
1. 复现问题，确认错误原因
2. 修复内容，验证修复有效
3. 更新CHANGELOG.md记录修复内容
4. 提交PR，说明问题和修复方案

### 任务2：更新内容
1. 确认更新内容的准确性，验证链接、命令有效
2. 更新对应文档
3. 涉及时效性内容更新最新动态区和对应表格
4. 更新CHANGELOG.md
5. 提交PR

### 任务3：新增文档
1. 确认文档所属分类，放到对应目录下
2. 按照统一格式编写文档
3. 在README的对应表格中新增文档链接
4. 更新CHANGELOG.md
5. 提交PR

### 任务4：版本对齐
1. 对照OpenClaw官方版本变更说明，更新所有涉及版本的内容
2. 在docs/version-changes/目录下新增对应版本的变更记录
3. 更新README中的版本号
4. 全量验证所有命令、配置是否符合新版本
5. 提交PR

## ❌ 禁止操作
1. **禁止修改核心项目文件**：`.gitignore`、`LICENSE`、`AUTHORS.md`、作者邀请链接等，除非明确要求
2. **禁止删除现有文档**：过时内容标注「已过时」即可，禁止直接删除
3. **禁止破坏目录结构**：新增文档必须放到对应分类目录下
4. **禁止替换作者信息**：所有作者相关内容、二维码、邀请链接必须保持原样
5. **禁止加入无关内容**：所有内容必须和OpenClaw教程相关，禁止加入广告、无关链接
6. **禁止提供未验证内容**：所有命令、配置必须测试验证有效才能发布
